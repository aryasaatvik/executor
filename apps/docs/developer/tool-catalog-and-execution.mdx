---
title: "Tool Catalog And Execution"
description: "How source materialization becomes callable tools and how invocation works."
---

Executor turns connected integrations into a workspace tool catalog.

That catalog is the thing the runtime actually executes against.

The execution sandbox is selected from `.executor/executor.jsonc`:

```jsonc
{
  "runtime": "quickjs"
}
```

Supported values are `quickjs` (default), `ses`, and `deno`.

## How tools are produced

At a high level:

1. an adapter materializes a source
2. the materialization writes documents, schema bundles, and operations
3. the control plane loads those persisted operations into a workspace tool index
4. the runtime invokes one tool at a time through the control plane

This means executor does not rebuild every integration from scratch on every execution.

## Why operations exist as a separate layer

The operation layer exists to support:

- search
- inspection
- policy
- fast tool lookup
- tool description without loading huge raw documents every time

Raw upstream documents are still kept, but they are not the best hot-path query surface.

## Schema transport

Executor uses compact operation-root schemas plus shared schema bundles.

This is better than storing expanded per-tool schema blobs because it avoids:

- massive duplicate schema payloads
- slow formatting and rendering
- treating presentation output as canonical storage

The result is:

- smaller payloads
- more reusable schema graphs
- cleaner tool detail responses

## Tool catalog shape

The tool catalog is built from four distinct sources:

### 1. System tools
Built-in discovery and introspection tools with `sourceKey: "system"`:
- `catalog.namespaces` - List available namespaces with display names and tool counts
- `catalog.tools` - List tools with optional namespace and query filters
- `describe.tool` - Get metadata and optional schemas for a tool path
- `discover` - Search tools by intent and return ranked matches

### 2. Executor tools
Workspace management tools with `sourceKey: "executor"`:
- `executor.sources.add` - Add an MCP, OpenAPI, or GraphQL source to the current workspace

### 3. Local tools
User-defined tools loaded from the `.executor/tools/` directory. These are transpiled TypeScript/JavaScript files with the `sourceKey` prefix `local.tool.*`. For example, a file at `.executor/tools/hello.ts` becomes tool path `local.tool.hello`.

### 4. Source catalog tools
Tools from connected sources (MCP servers, OpenAPI specs, GraphQL endpoints) that are persisted in the workspace SQLite database after authentication and schema extraction.

This four-source model lets executor present one unified discovery and invocation surface even though the underlying tools come from different places.

## Invocation flow

There are two distinct invocation paths based on tool source:

### Authored tool invocation (system, executor, local)

Tools from system, executor, and local sources are invoked directly without policy or auth checks:

1. Tool path is matched against the authored tool registry (system + executor + local tools)
2. Tool is invoked directly via `authoredInvoker` - NO policy lookup, NO auth resolution

This path is used for:
- `catalog.namespaces`, `catalog.tools`, `describe.tool`, `discover`
- `executor.sources.add`
- Any tool in `.executor/tools/` (prefixed with `local.tool.*`)

### Persisted tool invocation (source catalog)

Tools from connected MCP/OpenAPI/GraphQL sources go through full orchestration:

1. Resolve the tool by path from the workspace database
2. Run policy and approval checks via `authorizePersistedToolInvocation`
3. Resolve auth material for the source via `SourceAuthMaterial.resolve`
4. Load the manifest and schema bundle from the source catalog
5. Delegate actual invocation to the IR adapter (`invokeIrTool`)

This is the key point:

- the control plane orchestrates
- the adapter performs the protocol-specific call

### Tool precedence

The catalog merges sources in this order (later sources take precedence):
1. `local_user` - local tools from `.executor/tools/`
2. `persisted_source` - tools from connected sources (MCP, OpenAPI, GraphQL)
3. `executor` - executor tools like `executor.sources.add`
4. `system_helper` - system tools like `discover`

Note: The source catalog is being merged first in an upcoming change, so `getToolByPath` will find source tools before authored tools.

## Why this is flexible

This architecture is flexible because it keeps a stable middle:

- the runtime only needs tools and invokers
- the control plane only needs the canonical operation and auth model
- adapters can vary widely at the source boundary without changing the runtime contract

That is what makes it possible to add more source types without constantly rewriting execution logic.
